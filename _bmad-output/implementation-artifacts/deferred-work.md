## Deferred from: code review of 1-0-project-scaffolding-deployment-foundation (2026-07-14)

- **Backup writes to a local Docker volume, not off-host storage (AC5)** [docker/docker-compose.yml:92-115, docker/backup/pg_dump_daily.sh] — Reason: blocked on hosting provider decision. Off-host storage can't be concretely implemented until a hosting provider is chosen, which is itself an explicitly deferred decision elsewhere in this story. Revisit once hosting is selected.
- **GET /health reports liveness only, not DB readiness** [api/routes/health.py] — Satisfies AC4's literal "liveness endpoint" wording, but since this endpoint drives every container's `restart: always` healthcheck, a fully-down database currently wouldn't trip automatic recovery. Only matters once a real DB-backed route exists (Story 1.1+).
- **Nginx only proxies /health and /webhooks/ to the API** [docker/nginx/nginx.conf] — Any future backend route (e.g. Story 1.1's login endpoint) needs a new `location` block added by hand, or it silently falls through to the SPA's `index.html`. Not a bug today (only these two routes exist), but a sharp edge the next story should know about.
- **create_engine()/create_session_factory() have no singleton caching** [adapters/persistence/database.py:22-24] — Calling `create_engine()` more than once would create redundant connection pools. Currently unreachable (nothing calls it yet); whoever wires up the first real repository should cache it as a singleton.
- **Postgres bound to 127.0.0.1:5432 on the host in all three environments** [docker/docker-compose.yml:27-28] — Via the single shared compose file. If a staging/production host ever runs another Postgres instance on 5432, this stack fails to start. Contingent on the still-undecided hosting/infra layout.
- **init-roles.sh only runs once via docker-entrypoint-initdb.d** [docker/postgres/init-roles.sh] — A disaster-recovery restore onto a fresh volume from a `pg_dump` (schema/data only, no roles) needs this script re-run manually. Standard Postgres behavior, undocumented; matters once a DR runbook exists.

## Deferred from: code review of 1-2-first-run-administrator-bootstrap (2026-07-17)

- **Bootstrap endpoint has no defense against a hostile first caller — permanent, unauthenticated admin-creation race** [api/auth/routes.py:130-156, domain/bootstrap.py, adapters/persistence/users.py:76-80] — Reason: `POST /auth/bootstrap` defends against a racing second caller and later callers, but not an anonymous visitor reaching a fresh instance before the real operator; also the advisory lock has no timeout (connection-pool exhaustion risk under a flood of unauthenticated requests). Decision: mitigation is expected to come from deployment/network isolation (e.g. firewalling until first Administrator creation), not app code. Revisit if network-level isolation isn't feasible for a given deployment.
- **Username collision with a pre-existing non-Administrator user is unhandled (bare 500, not the app's error envelope)** [domain/bootstrap.py] — Reason: currently unreachable — the only way to create any `users` row today is via this same bootstrap endpoint (which locks itself after the first Administrator); Epic 3 ("Manage Users"), the only future path that could seed a colliding non-Administrator username, is still backlog. Revisit when Epic 3 ships.
- **`BootstrapForm.tsx` duplicates `LoginPage.tsx`'s layout and submit-handling logic near verbatim** [web/src/pages/BootstrapForm.tsx, web/src/pages/LoginPage.tsx] — Reason: this story's own Dev Notes discourage over-abstracting a one-time flow, and no third consumer exists yet to justify extraction. Revisit if a third similar form appears (e.g. Story 1.5's password reset).

## Deferred from: code review of 1-3-role-scoped-portal-access-last-admin-guard (2026-07-17)

- **No audit entry for role-based authorization rejection at `get_current_user`** [api/auth/dependencies.py:53-54] — Reason: belongs to Story 1.4 (revocation). Login-time role rejection (`domain/auth.py`) writes a `login.failure` audit entry via `AuthenticationService.login`; per-request role rejection at the shared `get_current_user` dependency writes nothing — a previously-issued, still-valid token for a user whose role no longer qualifies is silently 401'd with no forensic record. Revisit when Story 1.4 designs session/token revocation and its audit trail.
- **`LastAdministratorGuard.ensure_can_deactivate` has a check-then-act race, no locking** [domain/administrators.py:260-266] — Reason: reads `count_active_administrators()` and decides in a separate step, with no row lock/`SELECT ... FOR UPDATE`/advisory lock (contrast with `acquire_bootstrap_lock`'s explicit `pg_advisory_xact_lock` for the same class of problem in Story 1.2). Two concurrent deactivate requests against the last two active Administrators could both read `count == 2`, both pass, and both proceed — leaving zero active Administrators, the exact failure FR-2 exists to prevent. Currently unreachable (guard has no caller yet), and per this story's own Dev Notes the locking/transaction design belongs to Story 3.1 ("that endpoint's real shape... is Story 3.1's job to design"). Revisit when Story 3.1 wires up the deactivate/delete endpoint.
- **`count_active_administrators` query has no index on `role`/`status` columns** [adapters/persistence/users.py] — Reason: pre-existing schema (columns from Story 1.1's migration `98ddc369b175`, no new migration needed for this story); low risk at expected table size. Revisit if this becomes a hot path at scale.

## Deferred from: code review of 1-5-login-lockout-password-reset (2026-07-18)

- **`/auth/forgot-password` reopens the login-flow's timing side-channel** [domain/password_reset.py:55-86, api/auth/routes.py:246-276] — Reason: a valid username triggers extra DB work (INSERT token + INSERT audit row) before returning, unlike login's dummy-hash-equalized timing. Low exploitability for Phase-1 (admin-only, single-tenant, self-hosted deployment; precise timing measurement requires network access an external attacker is unlikely to have). Revisit if the deployment becomes multi-tenant or public-facing.
- **Password reset does not invalidate other already-issued JWT sessions for the account** [domain/password_reset.py:88-99] — Reason: deliberately out-of-scope per this story's own Dev Notes — no per-user session registry exists (`revoked_tokens` is keyed by individual `jti`, not by user); building bulk per-user revocation is out of scope for AD-11's schema and this story's ACs.
- **No rate limiting on `/auth/forgot-password` or `/auth/reset-password`** [api/auth/routes.py:246-295] — Reason: consistent with app-wide absence of throttling infrastructure; no rate-limiting middleware exists anywhere in this codebase today.
- **`increment_failed_login_count`'s `scalar_one()` would raise unhandled if the user row vanished mid-request** [adapters/persistence/users.py] — Reason: unreachable today; no user-deletion feature exists anywhere in the codebase (only test cleanup deletes user rows).
- **`password_reset_tokens.user_id` FK has no `ON DELETE` behavior** [alembic/versions/8ae7e5d0d8c9_login_lockout_and_password_reset.py] — Reason: only matters once a user-deletion feature exists; will hard-fail on any user with reset-token history at that point.
- **Concurrent simultaneous lockout-triggering requests can each independently call `lock_until` and write a duplicate `account.locked` audit row** [domain/auth.py:66-84] — Reason: narrow race window, results only in a duplicate audit row, no functional or security harm.
- **Requesting a new reset link does not invalidate prior outstanding tokens for the same user** [domain/password_reset.py:55-86] — Reason: not required by any AC; a defense-in-depth nicety beyond this story's scope.
- **`new_password`'s 72-character cap (not byte length) lets bcrypt's 72-byte truncation bite on multi-byte-character passwords** [api/auth/routes.py:70-72] — Reason: pre-existing pattern copied verbatim from `LoginRequest`/`BootstrapRequest` (Story 1.1), not newly introduced by this story.
- **`new_password` has no complexity/strength requirement** [api/auth/routes.py:70-72] — Reason: same pre-existing convention as every other password field in this app.
- **Frontend lockout countdown is seeded once and counts down client-side with no re-sync to the server** [web/src/pages/LoginPage.tsx:43-56] — Reason: cosmetic only; the server still enforces the real lockout independently, so clock drift can at worst re-enable the submit button slightly early, resulting in one extra rejected request.
- **`retry_after_seconds` is only in the JSON body, no standard `Retry-After` HTTP header** [api/auth/routes.py:97-105] — Reason: beyond what AC #1 requires; the body already carries the remaining seconds.
- **`PasswordResetService.token_ttl` is unused by `complete_reset`** (`reset_password` route passes a meaningless `timedelta(0)`) [domain/password_reset.py:41-53, api/auth/routes.py:286-288] — Reason: already flagged and accepted as "slightly awkward" in this story's own Dev Notes; a future refactor could split `token_ttl` into a `request_reset`-only parameter.
