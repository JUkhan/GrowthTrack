# Story 4.4: Daily Report Schedule Configuration

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Administrator,
I want to view and change the Daily Report's global send time,
so that I can adjust when the team receives it without needing engineering involved.

## Acceptance Criteria

1. **Given** the current Daily Report schedule **When** I open Settings **Then** I can see the configured send time

2. **Given** a new send time **When** I save it **Then** it becomes the schedule the next scheduler run uses, and the change is audit-logged co-transactionally

3. **Given** the Daily Report schedule **When** it is stored **Then** it is a single, global setting persisted in the `ReportSchedule` entity (Architecture spine AD-11), not an environment variable — no per-recipient customization exists in Phase 1, and no redeploy is required to change it

- [x] [Review][Patch] Interval-poll redesign creates orphaned `Notification` rows every day — the `daily_report` job now polls every `report_schedule_poll_interval_seconds` (default 60s), and `_should_run_daily_report` returns `True` on every tick once today's target time has passed. ## Dev Notes

- **This story is greenfield for `ReportSchedule`** — no migration, domain model, port, repository, route, or frontend page for it exists yet (confirmed via repo-wide search). `config.py`'s `report_send_cron_hour`/`report_send_cron_minute` (added ahead of time by Story 4.2, with an explicit comment: *"ReportSchedule (AD-11, DB-backed, Administrator-editable) is Story 4.4's job"*) are the env-var placeholder this story replaces, not a pattern to extend.
- **The core design decision is `cron` → `interval` for the scheduler job**, so a DB-value change takes effect without a container restart (AC #3's "no redeploy required"). This exactly mirrors Story 4.3's own `retry_failed_deliveries` job, which is already `"interval"` specifically so it can read fresh DB/config state on every tick rather than being fixed at process start. Do not try to dynamically reschedule/replace the APScheduler job's cron trigger at runtime (e.g. via `scheduler.reschedule_job`) when the DB value changes — that would require the API process to reach into the separate scheduler process's in-memory job store across a container boundary, which nothing in this architecture supports (AD-5 keeps `api/` and `scheduler/` as separate processes/containers). The interval-poll approach needs no cross-process signaling at all.
- **Storage is UTC, the Administrator-facing value is Asia/Dhaka** — same split `nightly_import_cron_hour`/`report_send_cron_hour` already use today, and the same rule epics.md's Additional Requirements states generally ("timestamps stored/transmitted as ISO 8601 UTC, converted to Asia/Dhaka only at presentation edges"). The conversion belongs at the `api/settings` boundary (Task 5), not in `domain/report_schedule.py` — the domain layer only ever sees UTC hour/minute.
- **No version/optimistic-concurrency column on `ReportSchedule`.** Story 3.4's conflict-detection column only applies to `User`/`Team`/`RecipientList` (multi-editor directory entities); `ReportSchedule` is a global singleton with no plausible concurrent-edit scenario worth building UI for in Phase 1 — same reasoning Story 4.5 already established for `MessageTemplate`. Don't add one speculatively.
- **Audit logging is required here, unlike the theme-preference precedent.** `domain/preferences.py`'s `UserPreferenceService` deliberately has *no* audit write (a personal display setting isn't in AD-7/FR-12's audited-action list) — do not copy that shape. `domain/preferences.py`'s own docstring already names "Daily Report schedule changes" as one of the actions that *is* audited; this story is what fulfills that forward reference. Follow `MessageTemplateDirectoryService`'s audit pattern (`domain/notifications.py:282-388`) instead: write the `AuditLogEntry` in the same service call as the schedule update, never a separate transaction.
- **Duplicate-run safety is already solved — don't rebuild it.** Once the interval job's `now >= today_target` check goes true, every later poll that same day will also be true, but `ScheduledReportService.run_daily_report`'s existing AD-2 partial-unique-index guard (already relied on today for scheduler-restart/cron-misfire duplicate protection) makes every call after the first a clean, cheap no-op. Building a second "already ran today" flag on `ReportSchedule` itself would be redundant with a mechanism this story doesn't need to touch.
