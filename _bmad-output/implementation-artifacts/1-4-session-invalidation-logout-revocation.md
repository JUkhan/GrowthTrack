# Story 1.4: Session Invalidation â€” Logout & Revocation

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Administrator,
I want to log out, and to be logged out automatically if my account is deactivated mid-session,
so that a session can always be invalidated per policy, not just left to expire naturally.

## Acceptance Criteria

1. **Given** an active session, **when** I log out, **then** a revocation record is written keyed by the session's JWT `jti`, **and** the shared auth dependency rejects that `jti` on any subsequent request, even before natural expiry. [Source: epics.md#Story 1.4]
2. **Given** an Administrator is deactivated while holding an active session, **when** their next action reaches the portal, **then** they are logged out with a plain-language explanation, not a silent redirect. [Source: epics.md#Story 1.4]

## Dev Notes

- **This story has a real (if intentionally minimal) frontend, unlike Stories 1.2/1.3.** Those were backend-only ("As the system" framing, no UI). This story's own framing is "As an Administrator, I want to log out" â€” a literal user action needs a literal button somewhere. Task 7 places it on `HomePage.tsx`, the only authenticated screen that currently exists, explicitly **not** at `EXPERIENCE.md`'s eventual avatar-menu/Settings location (which has no home yet â€” Story 1.6 builds the nav shell, Story 4.4 builds Settings' other content). Do not build a nav shell or Settings stub here just to place the button "correctly" â€” that's exactly the kind of forward-building Story 1.1's own Dev Notes warned against ("don't build a Dashboard placeholder that pretends to be the real thing"). Flag this placement as provisional for whoever builds Story 1.6.
- **AC #2's mechanism is a direct `user.status` check on every request, not a jti-revocation record written by a deactivation event.** The Architecture spine's AD-8 prose ("Logout and mid-session Administrator deactivation write a revocation record checked by that same dependency") reads as if deactivation itself writes a `jti`-keyed row â€” but no deactivate endpoint exists yet (Story 3.1, still `backlog`) to write one, and even once it does, it has no reliable way to know every `jti` currently in play for that Administrator (JWTs are stateless; nothing tracks "issued tokens per user"). Story 1.3's own review already flagged the practical fix and pointed it at this story: *"`get_current_user` never checks `user.status`... likely intended to be closed by Story 1.4's revocation check, but status/revocation sync isn't guaranteed as written."* Checking `user.status != UserStatus.ACTIVE` directly on the freshly-fetched user row (already loaded every request for the role check) satisfies AC #2 exactly and more robustly than a jti-list ever could â€” it catches *every* outstanding session for a deactivated Administrator immediately, not just ones some future endpoint remembered to revoke. Task 5 implements it this way; do not build a "deactivation writes revocation rows for all outstanding sessions" mechanism instead â€” there's no session registry to enumerate from, and AC #2 doesn't require one.
- **Why `get_current_session` wraps `get_current_user` (not the reverse).** AD-8 requires "one shared FastAPI dependency" â€” Task 5 keeps that true by making `get_current_session` the one real pipeline and `get_current_user` a one-line derived view (`return current.user`), rather than duplicating the pipeline in two places or changing every existing/future route's dependency type from `User` to a tuple. Only `POST /auth/logout` (needs the `jti`) depends on `get_current_session` directly.
- **Why the deactivation 401 gets a distinct message but the revoked-token 401 doesn't.** Story 1.3 established "always return the identical generic 401" specifically to prevent a response from becoming an oracle about *someone else's* account (wrong password vs. wrong role vs. nonexistent username, all from an unauthenticated caller's point of view). Both of this story's new checks fire only against a caller who already possesses a validly-signed token for a specific, already-known Administrator â€” there's no other-account information to leak. But UX-DR25/EXPERIENCE.md specifically require deactivation to be explained ("logged out... with a plain explanation, not a mysterious redirect") â€” logout-then-replay doesn't have the same explicit requirement and the cause is self-evident to whoever triggered it, so it stays generic. Don't generalize this into "always give a distinct message now" or "never give one" â€” the two checks are different in kind for this specific reason.
- **`revoked_tokens` has no cleanup/expiry-purge job.** Every logout adds one permanent row; nothing ever deletes them, even after the underlying JWT's `exp` has long passed and the row is no longer load-bearing for anything. Not a blocker (table stays tiny at Phase-1's single-Administrator-ish scale), but a real gap â€” flag it as deferred work during this story's review rather than silently building a purge job that's not asked for by either AC.
- **No new Alembic migration touches the `users` table.** `User.status` already exists (Story 1.1's migration `98ddc369b175`) â€” Task 5's status check is a new read of an existing column, not new schema. Only `revoked_tokens` is new (Task 3).
- **No Nginx/CI changes needed.** `docker/nginx/nginx.conf`'s existing `location /auth/` block (added in Story 1.1) is a prefix match â€” it already proxies `POST /auth/logout` with no further change.
