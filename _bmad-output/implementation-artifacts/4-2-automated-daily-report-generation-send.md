# Story 4.2: Automated Daily Report Generation & Send

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Administrator,
I want every configured Recipient to automatically receive a correctly formatted Daily Report once a day,
so that the team stays informed without anyone manually triggering it.

## Acceptance Criteria

1. **Given** the scheduled trigger time arrives (once per operational day) **When** the scheduler fires **Then** a Scheduled Notification is created, targeting all configured Recipients, and generation completes within 60 seconds

2. **Given** the Daily Report content **When** it is generated **Then** it matches `sample-whatsapp-report.md`'s format and figures (YTD/MTD sales, Achievement %, Growth %, team performance, condensed top/focus brand and doctor names, Cr BDT currency formatting) sourced from Epic 2's computed data

3. **Given** a Recipient reachable via more than one targeting mechanism (individual + Team, e.g.) **When** recipients are resolved for the day's run **Then** they are de-duplicated to exactly one `NotificationDelivery` row, keyed by `(recipient_user_id, operational_day)`

4. **Given** the Send Event uniqueness constraint **When** the scheduler restarts or fires twice for the same operational day **Then** no duplicate `NotificationDelivery` rows are created â€” the partial unique index rejects the duplicate

5. **Given** delivery begins **When** each recipient's send completes **Then** it lands in Notification History tagged "Scheduled" within 5 minutes of the scheduled time **And** the Dashboard's notification-status field (Story 2.2) reflects this run's outcome once it is the most recent send system-wide

## Dev Notes

- **This story is a lateral extension of Story 4.1's already-built hexagonal slice, not a greenfield one.** `Notification`/`NotificationDelivery`/`NotificationTarget`/`MessageTemplate` domain types, ports, adapters, and the two AD-2 partial unique indexes already exist and already correctly support `notification_type=scheduled` â€” Story 4.1's own migration deliberately built both index variants up front. The genuinely new work is: (a) making `created_by_user_id` nullable, (b) assembling Daily Report *content* from three existing, untouched Epic 2 read services, (c) an advisory-lock + unique-index-catch idempotency path distinct from Manual's picker-driven flow, and (d) scheduler wiring. Do not re-derive or duplicate anything Story 4.1 already built (recipient resolution, atomic claim, Twilio adapter, dashboard status tile) â€” extend it.
- **WhatsApp template parameters cannot contain newline characters, tabs, or 4+ consecutive spaces** (current Meta WhatsApp Business Platform template policy, verified 2026-07-21 â€” see Sources below) â€” this directly conflicts with `sample-whatsapp-report.md`'s literal multi-line layout (one team per line, one doctor per line) if a naive design tried to pack the whole "Team A : 45% / Team B : 50% / Team C : 40%" block into a single template variable with embedded `\n` characters. This is a real current platform constraint, not a stylistic choice. **Design decision for this story**: `team_performance` and `top_doctors` content-variable values use a non-newline separator (e.g. `"; "`) between entries instead of literal newlines â€” preserves all the same information (team names, percentages, doctor names, in the same order) but cannot reproduce the sample's exact line-broken visual layout inside a single Content API variable. This is a genuine, `[ASSUMPTION â€” CONFIRM]`-worthy divergence from a literal reading of AC #2 ("matches `sample-whatsapp-report.md`'s format") â€” the *figures and information* match; the *literal line-break layout* cannot, given the platform constraint discovered above. Flag this explicitly rather than silently picking a separator and calling AC #2 fully satisfied. An alternative some teams use â€” a fixed maximum number of individually-numbered slots (`team_1_line`, `team_2_line`, ... up to some N, blank-padded) â€” was considered and rejected: Meta also disallows empty-string parameter values, and the number of Teams is data-driven (currently 4 in demo seed data, not fixed by any spec), so a fixed-slot design would need silent truncation or a template redesign the moment a 5th Team is added â€” a separator-joined single variable scales to any Team/doctor count without a template change.
- **Territory-to-Recipient mapping (`Team.name == Doctor.territory`) is this story's own inferred convention**, not stated in any planning document as a rule â€” see Task 4's `[ASSUMPTION]` note for the full reasoning and the `scripts/seed_demo_data.py` evidence backing it. If a future Source System integration (AD-6, still unidentified per PRD Open Question #1) ever ingests `Team`/`Doctor` data where team names and territory strings diverge, this mapping silently breaks (empty doctor sections, not an error) â€” worth a code comment at the match site pointing back to this story for whoever debugs that later.
- **No portal UI work in this story** â€” unlike Story 4.1's Task 6/7 (composer page, Dashboard wiring), Story 4.2 has zero frontend tasks. The Dashboard's notification-status tile (`web/src/pages/DashboardPage.tsx`, wired in 4.1) already reads `GET /dashboard/notification-status`, which already reads `most_recent_status_summary()` ordered by `Notification.created_at` system-wide, `notification_type`-agnostic â€” a Scheduled run's outcome will show up there automatically with no frontend change, per AC #5's second clause. Story 2.4's own precedent (doctor visit list shipped with zero portal screen, verified only via domain/repository tests) applies again here for the same reason: no Daily-Report-viewing UI exists or is planned (Notification History is Epic 5 Story 5.1, not yet built) â€” AC #5's "lands in Notification History" is, like Story 4.1's AC #7, only satisfiable at the data layer in this story (correct `notification_type='scheduled'` tagging, queryable later).
- **No audit-log entry for a Scheduled run** â€” mirrors `domain/ingestion.py`'s explicit, already-justified precedent (system-triggered job, no human actor, AD-7's co-transactional rule governs *administrative actions*, not background jobs) almost exactly. Use structured JSON logging with a `correlation_id` per run instead, same shape as `domain/ingestion.py`'s `_run_pipeline` logging (`extra={"correlation_id": ..., ...}`) â€” `scheduler/main.py`'s `_JsonFormatter` already renders this correctly for any `extra=` fields, no logging-infra change needed.
- **Two distinct idempotency mechanisms, layered, not redundant**: the new advisory lock (Task 2/3) narrows the *concurrency* window (two processes/threads racing within the same trigger); the pre-existing AD-2 partial unique index (Story 4.1's migration) is the *authoritative* zero-duplicate-send guarantee for the case the lock doesn't fully cover (a scheduler container restart between the lock transaction's commit and a subsequent misfire re-trigger for the *same* operational day, potentially minutes or hours apart â€” well outside any single transaction's lock lifetime). Task 4 Step 6 must catch the resulting `IntegrityError` and treat it as an expected, loggable no-op â€” an *unhandled* crash on that path would technically still prevent the duplicate row (satisfying AC #4's literal text) but would leave the scheduler job failed/logged-as-an-error for a case that isn't actually an error.
- **Twilio dispatch-loop extraction (Task 4 Step 8) is a real, non-trivial refactor** of `domain/notifications.py:277-330` â€” read `ManualNotificationService.compose_and_send`'s existing implementation and its accompanying tests (`tests/domain/test_notifications_service.py`) fully before touching it. The claim-for-dispatch / broad-exception-guard / outcome-recording shape has its own code-review history (see Story 4.1's Review Findings â€” the "uncaught exceptions during dispatch can lose already-sent messages" patch) â€” preserve that exact behavior for both callers, don't regress it while extracting.

### Current WhatsApp Business Platform specifics (verified 2026-07-21, not just training data)

- Template body **parameter values** (the strings passed as `content_variables`) cannot contain newline characters, cannot contain more than 4 consecutive spaces, and cannot themselves start or end a template body. This governs the `team_performance`/`top_doctors` content-variable design decision above.
- Full URLs are disallowed inside body placeholders (anti-phishing) â€” not directly relevant to this story's fields (no URLs in the Daily Report), noted for completeness since it's a related constraint on the same parameter surface.
- These are Meta WhatsApp Business Platform template-approval rules, enforced at Meta's template-review stage in the Twilio/Meta console (the same out-of-scope-for-code approval process Story 4.1's Dev Notes already named for `MessageTemplate`) â€” nothing in this story's Python code can violate them since the code only ever supplies `content_variables` values, never the template body itself; the risk is entirely in how the *actual* Twilio Console "Daily Report" template gets authored, which this story's seed-script comment (Task 6) can only flag, not enforce.
