# Story 4.3: Delivery-Status Webhook & Automatic Retry

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Administrator,
I want failed sends to retry automatically and every attempt's outcome to be tracked accurately,
so that I can trust the delivery status shown without double-checking manually.

## Acceptance Criteria

1. **Given** Twilio sends a delivery-status callback **When** it hits `POST /webhooks/twilio/status` **Then** the request's signature is verified before any status update is applied

2. **Given** a status callback payload **When** its provider message SID doesn't match the `NotificationDelivery` row's current SID **Then** it is logged and ignored as a superseded/stale attempt

3. **Given** a status callback attempting to move status backward (e.g. Delivered → Queued) **When** it is evaluated **Then** it is rejected — status transitions are monotonic (`Queued → Sending → Delivered / Retrying / Failed`)

4. **Given** a send attempt fails **When** retry is eligible **Then** it is retried automatically for up to 3 additional attempts with exponential backoff (1 minute, 5 minutes, 15 minutes) `[ASSUMPTION: retry policy magnitude, pending confirmation — PRD §13.12]`, with every attempt logged regardless of outcome

5. **Given** a `NotificationDelivery`'s retries are exhausted within its Send Event **When** the final attempt fails **Then** it is left `Failed` and is not re-claimed until the next eligible Send Event (next operational day, or a fresh Manual Notification)

## Dev Notes

- **This story's two Send Event failure sources are unified through one decision function.** A delivery can fail either synchronously (the initial `WhatsAppSender.send_template_message` call raises, handled at send time by `ManualNotificationService`'s existing `_fail`) or asynchronously (Twilio accepts the send, then later reports `failed`/`undelivered` via the webhook). Both paths must land on the *same* `FAILED_RETRYABLE`-vs-terminal-`FAILED` decision (`_status_after_failed_attempt`, Task 4) so the retry budget is counted consistently regardless of which path discovered the failure. This story deliberately does **not** distinguish a permanently-invalid recipient (e.g. a malformed number, Twilio error 21211) from a transient carrier failure — both consume the same fixed attempt budget. A future story could special-case specific Twilio error codes as non-retryable; not required by any AC here.
- **`RETRYING` is a pre-existing, currently-unused `DeliveryStatus` enum value — leave it alone.** Story 4.1 added it (and `FAILED_RETRYABLE`) to the enum ahead of this story so `claim_for_dispatch`'s `WHERE status IN ('queued', 'failed_retryable')` SQL (already shipped, unchanged by this story) wouldn't need a later migration. This story's design uses `FAILED_RETRYABLE` as the actual "eligible for retry" state (matching that already-shipped SQL) — `RETRYING` has no code path that produces it, in this story or Story 4.1. Do not invent a use for it; it is not a gap this story needs to close. (Story 4.1's own `DashboardPage.tsx` already defensively maps it to a "warning" badge in case a future story starts using it — no frontend change is required here either way.)
- **No Dashboard/frontend changes in this story.** No AC above mentions the Dashboard tile. `NotificationDeliveryRepository.most_recent_status_summary`'s existing worst-status-wins aggregation (Story 4.1) already ranks `FAILED_RETRYABLE` at the same severity tier as `FAILED` — a still-retrying delivery correctly still shows as the Dashboard's worst-case tile today. A future story might want to visually distinguish "still retrying" (warning) from "exhausted" (error) on the Dashboard, but that's a product decision with no AC backing it here — don't build it speculatively.
- **Coordination with Story 4.2**: per `sprint-status.yaml`, Story 4.2 (Automated Daily Report) is queued immediately before this one and, as of this story's creation, has **not yet been implemented** — no `domain/scheduled_notifications.py`, no dispatch-loop extraction exists in the current codebase (`domain/notifications.py` still has the inline loop from Story 4.1 only). If Story 4.2 lands first (the expected order), this story's dev agent must extend whatever shared dispatch helper 4.2 built rather than duplicating claim/send/record logic a third time, and must make sure Scheduled deliveries also get `content_variables` persisted (Task 4). If this story is somehow implemented *before* 4.2, do not block on it — build the minimal helper described in Task 4/6, and leave a clear note for whoever implements 4.2 next to fold it in.
- **Webhook URL reconstruction is a real, easy-to-get-wrong detail** — see Task 5's reasoning. Using `str(request.url)` instead of the new `webhook_public_base_url` setting will pass locally (no reverse proxy in front of a bare `uvicorn` dev run) and fail silently in staging/production (every real Twilio callback gets rejected as an invalid signature, with no user-visible symptom other than delivery statuses never updating) — exactly the kind of gap that's invisible until it's in front of a real Twilio account.
- **AD-7's co-transactional audit rule does not apply to this story's writes** — same reasoning already established twice (nightly import, and Story 4.2's scheduled notifications if built): no human actor exists for either a webhook callback or a scheduler-triggered retry. Use plain `logger.info`/`logger.warning` calls (Task 6/7), not `AuditLogRepository`.

### Current Twilio specifics (verified 2026-07-22, not just training data)

- `MessageStatus` values for an outbound WhatsApp message: `queued`, `sent`, `delivered`, `undelivered`, `failed` (plus `read`, WhatsApp/RCS-specific, if the recipient has read receipts enabled — not required by any AC, mapped to `DELIVERED` here since it strictly implies delivery already happened). `undelivered` carries an `ErrorCode` (carrier-level delivery failure after Twilio accepted it); `failed` means Twilio itself rejected the send before it ever reached the carrier (account/Twilio-level, `ErrorCode` in the 2xxxx/3xxxx range). This story treats both identically (`WebhookOutcome.FAILURE`) — no AC asks for the distinction.
- Signature verification: `twilio.request_validator.RequestValidator(auth_token).validate(url, params, signature)`, using the `X-Twilio-Signature` header. Twilio's own guidance is explicit that this SDK-provided validator must be used rather than a custom HMAC implementation, and that the signed URL and parameter set must match *exactly* what Twilio POSTed (whitespace/missing-empty-value discrepancies break validation) — this is why Task 5 builds the URL from a dedicated `webhook_public_base_url` setting rather than trusting anything reconstructed from the request behind Nginx.
- SDK `twilio==9.10.9` (already pinned since Story 4.1) includes `twilio.request_validator.RequestValidator` — no new dependency.
