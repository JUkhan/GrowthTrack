- (decision) Working mode: Coaching path. Deliverable: spine only (build-substrate), no additional rendered artifact requested.
- (constraint) Inherited from SPEC.md/PRD (binding, not re-decided here): dashboard load <=3s; report generation <=60s; scheduled delivery <=5min; 500+ concurrent WhatsApp-recipient dispatch fan-out (not portal concurrency); HTTPS everywhere; passwords encrypted at rest; RBAC (Administrator-only portal, Sales User/Manager are WhatsApp-only, no portal auth for them); all admin actions audit-logged, same transaction as the action (FR-12); 99.5% uptime w/ auto-recovery; zero duplicate sends per Send Event with auto-retry; single-tenant, no multi-org.
- (constraint) Stack named by stack.md/PRD (candidate, to verify current versions before binding as AD seed): React + MUI frontend, FastAPI backend, PostgreSQL, SQLAlchemy + Alembic ORM, JWT auth, Twilio WhatsApp API (POC-only, production provider undecided).
- (question) Open questions PRD explicitly deferred to architecture: (1) Redis/Celery vs simpler mechanism for retry/scheduling/idempotency; (2) Recipient Group/Channel modeling (no entity today) + whether Team is standalone + whether Notification splits into template/history pair; (3) delivery-status feedback integration shape (webhook vs polling) - return path not in current one-directional workflow diagram; (4) Source System identity unknown - integration shape only, not the concrete system; (5) Twilio credential/secrets management; (6) production WhatsApp provider migration path (Twilio -> 360dialog/Gupshup/local reseller) - architecture should not hard-couple to Twilio; (7) deployment/environments/infra strategy - nothing decided yet.
- (decision) AD: Design paradigm = Hexagonal (Ports & Adapters). Binds: all. Prevents: WhatsApp-provider or Source-System-specific assumptions leaking into domain/report/metrics logic. Domain core (metrics, dedup, brand/doctor ranking, retry/scheduling orchestration) depends on nothing external; adapters (Twilio WhatsApp, Source System importer, Postgres repos, JWT) implement ports the core defines. Chosen because both WhatsApp provider and Source System are named by PRD as volatile/undetermined.
- (decision) AD: Scheduling/idempotency mechanism = Postgres-backed job/idempotency table + single in-process scheduler (APScheduler), NOT Redis/Celery for Phase 1. Binds: FR-6, FR-7, CAP-3, CAP-4. Prevents: two independently-built senders assuming different dedup/retry mechanisms. Rule: Send Event identity = unique(recipient_id, trigger_id, operational_day); enforced via a DB unique constraint, not just application-level checking. Sits behind the scheduling port so Celery can replace it in Phase 2 without touching domain code.
- (decision) AD: Delivery-status feedback = inbound webhook (Twilio status callback), not polling. Binds: FR-7, FR-11. New boundary: POST /webhooks/twilio/status, must verify Twilio request signature. Prevents: a poller and a webhook handler both existing and double-updating delivery status.
- (decision) AD: Data model - Recipient is a single targeting abstraction unifying Recipient Group and Recipient Channel (same fan-out-to-individual-numbers mechanism per PRD glossary, distinguished only by naming convention). Team is a standalone entity (independently CRUD'd per FR-9, used for both reporting rollup and targeting). Notification splits into a request/definition record and a per-recipient NotificationDelivery record (status, attempt count) - this is what the PRD's 'Send Event' concept implies. Binds: entities.md's User/Notification/Team gap, CAP-5, CAP-8, FR-9, FR-11.
- (decision) AD: Deployment = Docker Compose: API container, scheduler as its own container/process (isolated so a web-tier crash doesn't kill scheduled sends), Postgres, Nginx reverse proxy. Two environments: staging + production (local dev reuses the same compose file). Secrets (Twilio credentials, JWT signing key, DB credentials) via environment variables injected at deploy time, never committed; a dedicated secrets manager deferred past Phase 1. Bangladesh PDPA data residency NOT architected around now - flagged as a deferred legal-review risk per PRD Section 10, not a hard Phase-1 NFR.
- (decision) AD: Source System ingestion = contract only, not concrete system (system itself unidentified per PRD open question #1). Nightly batch-import adapter behind an ingestion port, landing into a staging table before upserting into SalesData/BrandPerformance/Doctor. Only the adapter changes when the real system is identified.
- (version) Verified current stable versions (2026-07-14, web-checked): FastAPI 0.139.0; React 19.2.7; MUI (@mui/material) v9.2.0; PostgreSQL 18.4 (v19 still beta, not chosen); SQLAlchemy 2.0.51 (2.1 still beta, not chosen); Alembic 1.18.5; Twilio Python SDK 9.10.9; APScheduler 3.11.3; PyJWT 2.13.0.
- (decision) Password hashing: pwdlib (bcrypt backend), NOT passlib - passlib is unmaintained and FastAPI's own docs now recommend pwdlib as of 2026. Caught via version verification, not asserted from training data.
- (decision) Naming convention: REST paths are plural-noun resources per stack.md precedent (/recipients, /teams, /notifications/history); DB tables snake_case, API schemas PascalCase; domain vocabulary is PRD-glossary-verbatim across all layers (Recipient, Send Event, Operational Day, Notification, NotificationDelivery) - no per-layer renaming of the same concept.
- (decision) Data & format conventions: all entity ids are UUIDv4 (cross-service correlation incl. Twilio webhook callbacks, not sequential ints). Timestamps stored/transmitted as ISO 8601 UTC always; conversion to Asia/Dhaka happens only at presentation edges (WhatsApp text, Dashboard, Operational-Day boundary calculation) - Operational Day is a presentation/business-logic concept, never a stored timezone. Money stored as fixed-precision decimal in base currency units; Cr BDT formatting is a presentation-edge concern only. Every REST error response uses one envelope shape: {error:{code,message,details}}.
- (decision) State & cross-cutting conventions: all writes to Recipient/Team/Notification/User go through the domain service layer, never direct repository access from route handlers. Every admin-mutating service method commits its data write and its Audit Log write in one DB transaction (FR-12 hard requirement - never one without the other). Every portal route authorizes via one shared JWT+Administrator-role dependency, never ad hoc per-route checks. Structured JSON logging with a correlation/request id threaded from inbound HTTP or scheduler trigger through to WhatsApp adapter calls. Config via one typed settings object (pydantic-settings) reading environment variables - no scattered os.environ calls.
- (decision) Entity model (names + relationships, ERD-level): User (Role: Administrator/SalesUser/Manager; OptInConsent state); Team (1 Team has many Users); RecipientList (unifies Group+Channel, has many Users via join table, Kind=Group|Channel for org convention only); Notification (request/definition: Type=Scheduled|Manual, target spec referencing Users/Teams/RecipientLists); NotificationDelivery (many per Notification, one per resolved-and-deduped individual User, carries status/attempt count - the 'Send Event' unit); SalesData (many per Team); BrandPerformance; Doctor (scoped by Territory); AuditLogEntry (references acting User); OptInConsent event history (per User, tracks number-change-invalidates-consent rule from FR-9).
- (decision) Source tree mirrors the hexagonal paradigm: api/ (inbound HTTP adapter, FastAPI routers incl. webhook receiver), domain/ (core services: metrics, dedup, ranking, notification orchestration - no external deps), ports/ (abstract interfaces domain depends on), adapters/whatsapp_twilio, adapters/source_system, adapters/persistence (SQLAlchemy repos implementing domain ports), scheduler/ (APScheduler jobs, own process entrypoint separate from the web process per the deployment AD), web/ (React app, separate top-level dir), alembic/, tests/, docker/.
- (decision) Capability to architecture map: CAP-1 (JWT auth) -> api/ auth routes + ports/auth, AD on shared JWT+role dependency. CAP-2 (Dashboard) -> api/dashboard route + domain/metrics service, reads SalesData/BrandPerformance. CAP-3 (auto WhatsApp report) -> scheduler/ job + domain/notification orchestration + adapters/whatsapp_twilio, governed by scheduling AD and Send Event uniqueness rule. CAP-4 (manual notification) -> api/notifications route + same domain/notification orchestration + adapters/whatsapp_twilio, same dedup rule. CAP-5 (recipient directory mgmt) -> api/recipients route + domain service over User/Team/RecipientList, audited via cross-cutting transaction rule. CAP-6 (brand analytics) -> domain/metrics ranking logic, consumed by both CAP-2 and CAP-3/4. CAP-7 (doctor list) -> domain/metrics doctor-ranking logic, consumed by CAP-3/4 only (Daily-Report-only per PRD). CAP-8 (notification history) -> api/notifications/history route reading NotificationDelivery. Audit Log (FR-12, cross-cutting) -> AuditLogEntry writes co-transactional with every mutating service call.
