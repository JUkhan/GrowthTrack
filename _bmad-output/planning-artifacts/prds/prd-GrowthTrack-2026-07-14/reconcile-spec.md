# Reconciliation — SPEC.md vs PRD

## Covered

- **CAP-1 (Admin JWT login)** — FR-1, §4.1; consequences match SPEC's success criterion (unauthenticated requests rejected, session TTL/invalidation policy) and correctly carry the TTL detail forward as an open question (§13 OQ#10) rather than inventing one.
- **CAP-2 (Dashboard, 7 fields, 3s load)** — FR-3; all seven named fields present, 3-second constraint preserved, formula ambiguity correctly tagged `[ASSUMPTION]` and cross-referenced to §8/§14.
- **CAP-3 (Automated daily report, retry, logging)** — FR-6 + FR-7; format-match to `sample-whatsapp-report.md`, 60s generation, 5-min delivery, retry-and-log behavior, no-duplicate guarantee all present, and reinforced by SM-1/SM-3/SM-C1.
- **CAP-4 (Manual notification)** — FR-8; select recipients, compose, optional report attach, immediate send, appears in history — all present, plus a concrete journey (UJ-1) and edge case (partial-failure-doesn't-block-group) not in SPEC's own prose.
- **CAP-5 (Recipient directory management)** — FR-9; individuals/groups/channels/teams CRUD, effect on future notifications, audit-logged — all present.
- **CAP-6 (Brand performance lists)** — FR-4; top/low-performing/focus brands, computed from dataset, available to both Dashboard and Daily Report. Ranking-threshold ambiguity correctly flagged as a PM decision rather than silently engineered around.
- **CAP-7 (Doctor visit list)** — FR-5; name/territory/target-priority fields, scoped per territory, explicit "no patient health data" reinforcement (SPEC assumption promoted into a testable consequence — a strengthening, not a weakening).
- **CAP-8 (Notification history)** — FR-11; date/time/recipient/type/status, same-operational-day accuracy, plus a reasonable filterability addition flagged `[NOTE FOR PM]` as unspecified-by-source.
- **Constraints** — all ten SPEC constraints (3s dashboard, 60s generation, 500+ concurrency, 5-min delivery, HTTPS, encrypted passwords, RBAC, audit logging, 99.5% uptime, no-duplicate/auto-retry) are traceable into §8 Cross-Cutting NFRs, §12 Operational Requirements, and the relevant FRs. The RBAC constraint — which sits in genuine tension with CAP-1's "Administrator only" framing — gets the most careful treatment in the whole reconciliation: FR-1/FR-2 resolve it, §4.1 flags the resolution inline as `[ASSUMPTION]`, and addendum §A5 preserves the full rationale. This is the single hardest reconciliation point in SPEC.md and it is handled thoughtfully rather than flattened.
- **Non-goals** — AI forecasting deferral (with Phase 2 pointer), the full future-enhancements list (charting, image-rich messages, Power BI, email/push, native app, scheduled customization, multi-language, export), and Twilio-POC-only/production-BSP-deferral are all present in §5, with the same items cross-referenced in §6.2 MVP Scope. PRD also adds two well-reasoned non-goals SPEC didn't spell out explicitly (SSO/identity, direct HCP messaging) — consistent extrapolations, not scope drift.
- **Success signal** — the SPEC's compound success statement (accurate/on-time report, zero duplicates, admin can trigger/retarget/audit without touching source systems) is distributed correctly: on-time/zero-dup → SM-1/SM-3; trigger → FR-8; retarget → FR-9 + UJ-3 (the "correct a stale phone number, next send picks it up" journey is a direct dramatization of "retarget"); audit → FR-11/FR-12; "without touching source systems" is verbatim in the Administrator JTBD (§2.1).
- **Assumptions** — pharma/FMCG domain and doctor-targeting rationale carried into Vision (§1) and JTBD (§2.1); "Doctor = HCP, not patient, no patient health data" is carried into the Glossary, FR-5, and §9 Privacy (strengthened into a testable consequence, appears three times).
- **Open questions** — 9 of SPEC's 10 open questions are either carried forward into PRD §13 with clarifying detail (source-of-truth system, report schedule/customization, brand/doctor data refresh cadence — bundled with #1, formulas, PDPA residency, BSP choice, Redis/Celery) or explicitly resolved with a stated rationale and an `[ASSUMPTION]`/decision tag (managers'-portal-access question → resolved via addendum §A5; OAuth2-vs-JWT → resolved in §14; pharma promotional-compliance → resolved in §14 with re-examination trigger noted).

## Gaps

- **SPEC open question "Are users created manually, or synced from an existing system (AD/HRIS)?" is silently resolved, not flagged.** FR-9's second consequence states as flat fact: "User accounts are created manually by an Administrator in Phase 1 — no external directory sync." This directly answers a SPEC open question but carries no `[ASSUMPTION]` tag and has no corresponding entry in §14 Assumptions Index or §13 Open Questions — breaking the PRD's own stated convention (§0: "inferred details are tagged `[ASSUMPTION]` inline and indexed in §9 [§14]"). Should land as an `[ASSUMPTION]` tag on that FR-9 bullet plus a §14 entry stating the no-sync decision and its rationale (or lack thereof).
- **SPEC open question "How exactly are users mapped to WhatsApp groups, channels, or individuals for targeting?" is dropped entirely.** It does not appear in PRD §13's eleven open questions, and no FR addresses the underlying mechanism (e.g., is a User's WhatsApp identity a phone number captured at creation, does adding a Recipient Group/Channel require the Administrator to already be a member/owner of that WhatsApp entity, how is a group's WhatsApp-side ID captured). FR-8/FR-9 cover CRUD on the directory but not this mechanics question. Should land as a new §13 open question, deferred to architecture/`entities.md` the same way OQ#7 (Team entity structure) already is.
- **The Bangladesh-locale assumption loses its stated evidence.** SPEC explicitly flags deployment locale as an assumption and cites concrete supporting evidence: the sample report's "Cr BDT" (crore taka) currency formatting. The PRD absorbs "Bangladesh" as ambient fact scattered across multiple sections (Asia/Dhaka default schedule in FR-6/§14, PDPA discussion in §10, BDT-reseller mention in addendum §A2) but never states the locale assumption itself as a single flagged item, and the "Cr BDT" evidence that justified it is not mentioned anywhere in prd.md or addendum.md. Since FR-6 requires the report to "match `sample-whatsapp-report.md`," the currency-format assumption is inherited only implicitly. Should land as a §14 Assumptions Index bullet stating the locale assumption and its evidentiary basis, so a reader of the PRD alone (without re-deriving it from the companion sample file) understands why Asia/Dhaka and BDT-related decisions appear elsewhere.
