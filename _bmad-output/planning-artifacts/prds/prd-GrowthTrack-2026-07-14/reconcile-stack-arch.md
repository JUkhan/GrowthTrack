Inputs: `_bmad-output/specs/spec-growthtrack/stack.md`, `_bmad-output/specs/spec-growthtrack/architecture-diagrams.md`

## Covered

- The full application stack (React + Material UI, Python FastAPI, PostgreSQL, SQLAlchemy + Alembic) is correctly omitted from the PRD entirely — zero references anywhere in `prd.md`, exactly matching stack.md's own framing as "implementation-shape reference for architecture/dev, not part of the WHAT contract."
- Twilio WhatsApp API is consistently and correctly cited as the Phase 1 POC-only provider (Glossary "WhatsApp BSP," §5 Non-Goals, §9 Cost, §10 Data Governance, §11 Integration and Dependencies), with the production-provider decision explicitly deferred to addendum §A2 and Open Question #5 — matches stack.md's "Twilio WhatsApp API (POC only — see Non-goals... for production-provider decision)" verbatim in spirit.
- stack.md's flagged Phase-2 open question — "Whether any of that infrastructure [Redis/Celery/AI-ML] is actually needed earlier, in Phase 1, is an open question" — is faithfully preserved as PRD Open Question #6, and §6.2 MVP Scope explicitly calls out background-job infrastructure as "an architecture-level implementation choice, not a PRD-level scope item." This is the PRD doing exactly the right thing: stating the WHAT-level guarantee (retry/scheduling/idempotency) without prescribing the mechanism.
- All 8 example REST endpoints in stack.md (POST /login, GET /dashboard, GET /reports/daily, POST /notifications/send, GET /notifications/history, GET /brands, GET /doctors, GET /teams) map cleanly onto a PRD FR (FR-1, FR-3, FR-6, FR-8/FR-6, FR-11, FR-4, FR-5, FR-9/FR-3 respectively) — nothing in that example endpoint surface was silently dropped, and the PRD correctly avoids restating the endpoints themselves as a WHAT-level artifact.
- The architecture-diagrams.md workflow shape (Sales Database → Backend API → Generate Daily Metrics → Prepare WhatsApp Message → Twilio WhatsApp API → Recipients) is not contradicted anywhere: FR-4's "computed once, consumed by both the Dashboard and the Daily Report so the two never disagree" matches the shared "Generate Daily Metrics" step; FR-6's generate-then-send flow matches the pipeline order; the Twilio hop is consistent throughout.
- Phase 2 pointers in both source files (roadmap-phase2.md for the AI/ML stack and the forecasting workflow variant) are correctly mirrored by the PRD's Vision (§1, "AI-driven forecasting... is real, wanted, and fully scoped in roadmap-phase2.md") and §5 Non-Goals, without scope-creeping forecasting into Phase 1 FRs.
- stack.md's WhatsApp interface functions "Send Message" and "Delivery Status" are functionally represented (FR-6/FR-7 for send + retry, FR-11 for delivery status in history) without the PRD naming Twilio's actual API methods — appropriate WHAT-level abstraction.

## Gaps

- **(a) Over-prescription, soft:** "JWT" is named explicitly and repeatedly (FR-1, FR-2 context, §8 NFR Security, §14 Assumptions Index) rather than described at the capability level ("token-based, expiring authenticated session"). stack.md lists "Authentication: JWT" as implementation-shape-only. The PRD's own Assumptions Index defends this as inherited from the source SPEC's CAP-1 (not from stack.md), which is a reasonable justification — but the PRD never makes that provenance distinction explicit inline, so a reader comparing against stack.md would reasonably flag it as the one piece of stack.md that leaked into WHAT-level FR language. Low severity; consider a one-line footnote clarifying "JWT" here reflects a prior SPEC-level commitment (CAP-1), not a stack.md-derived implementation choice.
- **(b) Shape gap, moderate:** The architecture-diagrams.md workflow is strictly one-directional (Sales Database → ... → Twilio → Recipients) with no return path shown. But FR-7 (delivery retry/failure logging) and FR-11 (delivery status in Notification History) both depend on delivery-status feedback flowing back from Twilio into GrowthTrack — which requires a callback/webhook or polling path the source diagram doesn't depict. §11 Integration and Dependencies states only "Messaging: Twilio WhatsApp API for Phase 1" with no mention of the inbound status-feedback dependency. Worth a one-line addition to §11 (or a new Open Question) noting that a delivery-status callback path is a required integration point not shown in the source workflow diagram, so architecture doesn't miss it when translating the linear diagram into a real system design.
- **(c) Dropped detail, minor:** stack.md's WhatsApp interface section lists three functions — Send Message, Delivery Status, **Authentication** — but the PRD's §11 Integration and Dependencies doesn't enumerate what's required of the Twilio integration beyond "Messaging: Twilio WhatsApp API." The "Authentication" function (Twilio API credential/secret handling to call the API) isn't referenced anywhere as a dependency. Likely architecture-level (credential/secrets management), so may not need a full Open Question — but a one-line Integration/Dependency note ("GrowthTrack backend requires secure storage/rotation of Twilio API credentials") would close the traceability gap back to stack.md.
- **(c) Consistency note, minor:** The PRD's Assumptions Index / §11 assumes the Source System is "an external ERP/CRM (name TBD), feeding GrowthTrack via nightly batch import," while architecture-diagrams.md's workflow labels the origin node only "Sales Database" — no indication it's external, no stated cadence (batch vs. real-time). Open Question #1 already covers "what system is the source of truth," which is good, but it's phrased around ERP/CRM identification rather than acknowledging the diagram's more neutral "Sales Database" framing. Consider broadening OQ1's phrasing slightly so architecture isn't anchored to "external ERP/CRM" before the source system is actually confirmed.
