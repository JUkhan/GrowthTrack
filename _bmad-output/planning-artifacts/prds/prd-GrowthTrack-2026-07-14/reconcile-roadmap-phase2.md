# Reconciliation — roadmap-phase2.md vs PRD

## Covered

- **Deferral is explicit, not silent.** PRD §5 Non-Goals bullet 1 names all six Phase 2 capabilities verbatim (monthly/territory/brand-demand/target-achievement prediction, doctor potential scoring, low-sales alerts), states they are "out of scope for Phase 1," and cites `roadmap-phase2.md` as where they are "fully scoped." PRD §1 Vision reinforces this with explicit framing ("AI-driven forecasting... is real, wanted, and fully scoped in `roadmap-phase2.md`... Shipping the reporting-and-management core cleanly is what makes Phase 2 possible, not a consolation scope cut") — this is a positive, intentional deferral narrative, not an omission.
- **Redis/Celery open question carried forward, left open.** roadmap-phase2.md flags "Whether Redis/Celery ... [is] Phase 2-only ... or should move earlier into Phase 1 is an open question — see SPEC.md." SPEC.md's Open Questions contains the same item. PRD §13 Open Questions #6 carries it forward near-verbatim ("Is dedicated background-job infrastructure (e.g., Redis/Celery) needed to meet the retry/scheduling/idempotency requirements (FR-6, FR-7), or does a simpler mechanism suffice? Deferred to architecture.") and PRD §6.2 reiterates it is explicitly not a PRD-level commitment either way. Correctly preserved as unresolved.
- **OAuth2 open question carried forward, resolved with traceable rationale (not dropped).** The same roadmap-phase2.md line also flags OAuth2 as unresolved-in-v2. SPEC.md's Open Questions asks "Is OAuth2 required for Phase 1 authentication, or is JWT alone sufficient until Phase 2?" The PRD does not ignore this — it closes it in §14 Assumptions Index ("OAuth2 is not required for Phase 1; JWT-based authentication alone is sufficient. Resolves a SPEC open question; consistent with CAP-1's explicit JWT commitment"), and FR-1's text commits only to JWT. This is grounded in SPEC.md CAP-1's actual wording ("JWT-based authentication," no OAuth2 mentioned), so the closure is defensible and traceable rather than a silent drop.
- **Team / Notification-split open question carried forward.** roadmap-phase2.md's "Additional Data Entity" section proposes `Team` as standalone and `NotificationLog` split from `Notification`; entities.md's Notes section frames this as an open modeling question. PRD §13 Open Question #7 carries it forward: "Does `Team` need to be a standalone data entity ... and should `Notification` split into a template/history pair? Deferred to architecture — see `entities.md`."
- **No Phase-2-only capability leaked into a Phase 1 FR.** Checked all twelve FRs: FR-1/FR-2 commit only to JWT (no OAuth2 in the FR text itself, only in the Assumptions Index as a closed question); FR-6's report content is scoped to `sample-whatsapp-report.md`, which contains no forecast/prediction section (confirmed by reading that file — it's YTD/MTD/Achievement/Growth/team/top-brand/focus-brand/top-doctors only, no forecast row); no FR references AI/ML libraries, Redis, Celery, `GET /forecast`, or the AI pipeline steps. FR-5's "Target Priority" doctor ranking is a static/manual field sourced from `entities.md`'s v1 Doctor entity (DoctorID, Name, Territory, Priority) — distinct from and not conflated with Phase 2's AI-driven "Doctor potential scoring."
- **Phase-2-only NFRs, tech stack, endpoint, and pipeline are absent from Phase 1 content**, consistent with the blanket Non-Goals deferral: "API response < 500 ms (excluding analytics)" and "Horizontal scalability" do not appear in PRD §8 Cross-Cutting NFRs; Scikit-learn/LightGBM/XGBoost/Prophet, Redis, Celery, OpenAPI/Swagger, Docker/Nginx/Gunicorn/Uvicorn do not appear as Phase 1 commitments; `GET /forecast` and the AI pipeline (extract → feature-engineer → train → forecast → store → display → WhatsApp summary) are absent from the PRD's feature set (§4) and report format (FR-6).

## Gaps

- **Additional/changed Phase 2 NFRs aren't individually triaged as Phase-1-relevant-or-not.** roadmap-phase2.md lists "API response < 500 ms (excluding analytics)" and "Horizontal scalability" under NFRs "per v2" without question-marking them (unlike the Redis/Celery/OAuth2 line, which explicitly says "is an open question"). The PRD relies on the blanket §5 Non-Goals reference to `roadmap-phase2.md` to cover these, but never states outright whether a general (non-analytics) API-latency target was ever intended for Phase 1's reporting/notification endpoints — Phase 1's §8 NFRs only specify dashboard load (3s) and notification generation (60s), not a general API response-time SLA. Low severity: plausible the PRD's silence is intentional (v2's NFR table is framed as additive to the whole stack including Phase 1 endpoints), but it was never explicitly confirmed either way, so a stricter reconciliation might add this as an Open Question rather than leave it implicit.
- **Non-Goals bullet itemizes capabilities but not the other roadmap-phase2.md sections by name.** PRD §5's forecasting bullet names the six candidate capabilities and the roadmap file, but doesn't separately name-check the `Forecast` entity, the AI/ML tech-stack additions, or the `GET /forecast` endpoint as deferred items. This is covered by the blanket "fully scoped in `roadmap-phase2.md`" reference and is not a real omission — flagged only as a minor explicitness/completeness nit, not a functional gap.
