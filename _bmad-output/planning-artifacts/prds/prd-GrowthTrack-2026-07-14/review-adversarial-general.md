# Adversarial Review: GrowthTrack Phase 1 PRD + Addendum

- **Recipient Group / Recipient Channel targeting has no verified technical feasibility path.** FR-6, FR-8, and FR-9 all treat "WhatsApp Group" and "WhatsApp Channel" as addressable send targets on the same footing as an individual User, and the addendum's entire cost model (§9, addendum A2) is built around Meta's per-delivered-template-message billing for business-initiated 1:1 conversations. Twilio's WhatsApp product (the confirmed Phase 1 provider) sends template messages to individual phone numbers — it does not send to WhatsApp Groups, and WhatsApp Channels use a separate, more restricted broadcast mechanism that doesn't fit the template-billing model described. Nothing in the PRD or addendum acknowledges this gap, verifies it against Twilio's actual API capabilities, or describes how "sending to a Group/Channel" would actually work (fan-out to individual members? a different API entirely?). This is a foundational scope item resting on an unverified assumption, not a minor implementation detail.

- **Vision (§1) and Non-Goals (§5) both claim Phase 2 forecasting is "fully scoped in `roadmap-phase2.md`," but that file explicitly says otherwise.** `roadmap-phase2.md` labels its own capability list "candidate, not yet validated as CAPs," leaves the `Forecast` entity's fields unspecified, and flags Redis/Celery and OAuth2 as unresolved. Calling this "fully scoped" is an overstatement the source document itself contradicts — the PRD should say "captured" or "preserved," not "fully scoped."

- **The "500+ concurrent users" NFR (§8, §12) is never reconciled with the Administrator-only portal decision.** Phase 1 restricts portal login to Administrators (§2.2, §4.1, addendum A5) — a role that, for a field sales org's back-office, is plausibly a handful to a few dozen people. A concurrency target of 500+ reads like an unexamined holdover from the original SRS (written when Sales Reps/Managers were assumed to log in too) that wasn't revisited after RBAC was narrowed to Administrator-only. No stated admin headcount or rationale backs the number.

- **No org-scale figures appear anywhere in the document.** Rep count, manager count, administrator count, doctor count, territory count, and expected daily message volume are never stated, even approximately. This makes it impossible to sanity-check the concurrency NFR, the cost projections (§9 says cost "scales with recipient count × send frequency" but gives no recipient count), or the manual-only account creation decision below.

- **FR-9's "accounts created manually by an Administrator, no external directory sync" is flagged as a confirmed decision but never stress-tested against scale.** If the field force runs into the hundreds (typical for pharma/FMCG), one-by-one manual entry through a web form — with no bulk import/CSV upload mentioned anywhere in scope — is an operational burden the PRD doesn't acknowledge, let alone justify.

- **The Dashboard's "seven fields" (FR-3) and the Brand/Doctor features (FR-4, FR-5) contradict each other on what actually renders on the Dashboard.** FR-4's consequences state brand lists are "available to both the Dashboard (FR-3) and the Daily Report (FR-6)," but FR-3's enumerated seven fields (Today's Sales, YTD, MTD, Achievement %, Growth %, team performance, notification status) don't include brand or doctor data at all, and FR-5 only ties the doctor list to the Daily Report, not the Dashboard. Either FR-3's field list is incomplete or FR-4/FR-5's claims about Dashboard availability are wrong — as written, both can't be true.

- **Manual Notifications (FR-8) have no retry/failure-logging guarantee, but UJ-1's edge case implies one exists.** FR-7 (retry & failure logging) sits under §4.5 Automated Daily Reporting and its "Realizes" line cites only UJ-3, not UJ-1. Yet UJ-1's edge case — "that single delivery is logged as failed with a retry attempt recorded" — describes exactly this behavior for a *manual* notification. FR-8's own consequences say nothing about retries. The journey narrative promises behavior no FR actually commits to.

- **Manual notifications have no latency SLA at all.** UJ-1 narrates "the message reaches every Team B recipient within minutes, well before the next scheduled run," but FR-8's consequences only say the message "reaches selected Recipients without waiting for the next scheduled run" — no time bound, unlike FR-6's explicit 60-second generation / 5-minute delivery targets for scheduled reports. The flagship "urgent update" journey has no measurable commitment behind it.

- **Editing a recipient's phone number (FR-9, exercised in UJ-3) doesn't address opt-in consent continuity.** WhatsApp opt-in is tied to a phone number, not a person. If an Administrator corrects a stale number, FR-9 says the fix "takes effect from the next send" — but nothing says whether the new number requires fresh opt-in capture (FR-10) before delivery resumes, or whether consent silently carries over. Given the addendum treats opt-in as a launch-blocking compliance requirement, this interaction should be explicit, not implied.

- **It's unclear whether opt-in/opt-out state changes land in the Audit Log.** FR-12 scopes the Audit Log to "create/edit/delete action on Recipients, Groups, Channels, or Teams" plus login events. Toggling a recipient's opt-in/opt-out status (FR-10) is arguably distinct from a directory "edit" and is never explicitly named as an audited action — a gap worth closing given the Vision's own framing of the audit trail as proof "that every notification went out correctly."

- **"Growth %" is never pinned to a specific metric per surface.** The Glossary defines it as "Month-over-Month or Year-over-Year, depending on context," but neither FR-3 (Dashboard) nor FR-6 (Daily Report) states which one displays where. The sample report shows only "MTD Growth," implying MoM, but the PRD never says the Dashboard or Daily Report exclusively uses MoM — leaving a field that's supposedly "testable" undefined in practice.

- **Several FRs are labeled "Consequences (testable)" while depending on values the document itself marks as unknown.** FR-3's Achievement %/Growth % formulas are `[ASSUMPTION, pending confirmation]`; FR-4's brand-tier thresholds are explicitly undefined pending a business decision; FR-6's schedule time is an assumption pending confirmation. Calling these consequences "testable" overstates the document's own rigor — they're testable only once inputs the PRD admits it doesn't have are supplied.

- **Compliance reasoning is applied with inconsistent rigor between addendum A3 and A4.** A4 (data residency) is treated appropriately as an open legal question requiring counsel, with the PRD correctly declining to guess. A3 (pharma promotional-compliance codes) reaches a confident conclusion — "do not directly apply" — based on the PM's own reasoning about message recipients, with no legal or regulatory citation and no "confirmed by user" tag (unlike other decisions in the doc that are explicitly marked as user-confirmed). A determination this consequential shouldn't read as more settled than the residency question it sits next to.

- **The data residency risk (§10) is deferred without a hard checkpoint or contingency plan.** It's flagged as "pre-launch legal review," but "pre-launch" is undefined relative to POC vs. production milestones, and there's no stated fallback if legal review lands after Twilio/Meta infrastructure decisions are already baked into the build. A finding that could "reshape the production WhatsApp provider decision" deserves a named gate in the plan, not just a footnote saying it won't block build work.

- **The Cost NFR (§9) describes the shape of the cost driver but commits to no number.** "Cost scales with recipient count × send frequency" is true of any messaging system and isn't a constraint — there's no budget ceiling, no estimated monthly spend, no approval threshold, and no volume assumption to compute one from. As written, this section can't be used to catch a cost overrun because it never defines what "over" would mean.

- **SM-1's "delivery success rate" conflates message handoff with actual delivery, and depends on infrastructure outside GrowthTrack's control.** A ≥99% target on delivery "within 5 minutes" bakes in Twilio/Meta/recipient-carrier reliability that GrowthTrack doesn't own, without acknowledging that a shortfall here may not be a GrowthTrack defect at all — the metric doesn't distinguish "we failed to send" from "WhatsApp/the recipient's phone failed to deliver."

- **SM-3 and SM-5 are QA acceptance criteria dressed up as success metrics.** "Duplicate-send rate: target 0" and "history/audit completeness: target 100%" are correctness bars any competent implementation should meet by construction — they don't tell a stakeholder anything about whether the product is succeeding, only whether it's not broken. Padding the Success Metrics section with restated FR consequences dilutes the metrics that actually measure value (SM-1, SM-4).

- **SM-4 (manual notification adoption) is statistically meaningless at the expected Administrator headcount.** With no stated admin population size (see above) and Phase 1 likely running with a handful of Administrators, a "% of Administrators who send ≥1 manual notification/month" metric will swing wildly on one or two people's behavior and won't validate anything reliably.

- **FR-6's "exactly one report within 5 minutes" and FR-7's open-ended retry policy are in latent tension.** Retry attempt count and backoff are an explicit open question (§8, OQ-12). If retries are allowed to run past the 5-minute window — which a reasonable backoff policy might do — then FR-6's hard per-recipient delivery SLA is violated by design for any recipient whose first attempt fails, yet nothing reconciles the two requirements or defines what "success within 5 minutes" means for a retried send.

- **§12 states a 99.5% uptime target but explicitly punts on the support/on-call model and RTO/RPO that would make that target meaningful (OQ-10).** An uptime number without a recovery-time commitment behind it is a headline figure, not an operational requirement — worth noting since the PRD elsewhere is careful to avoid exactly this kind of unbacked claim.

- **The Vision section leans on unearned marketing language.** Phrases like "closes that gap by delivering that visibility automatically" and "turning a reporting tool into something closer to a daily briefing" read as aspirational copy rather than commitments, and sit oddly next to a document that is otherwise disciplined about tagging assumptions and open questions. It doesn't break anything, but it's the kind of filler a skeptical stakeholder will call out as saying nothing.
